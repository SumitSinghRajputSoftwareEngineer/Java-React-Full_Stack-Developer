# MDC (Mapped Diagnostic Context)

> **Goal:** Understand MDC (Mapped Diagnostic Context), why it is important in modern applications, how it works internally, request correlation, user tracking, transaction tracking, practical examples, production use cases, and interview concepts.

---

# 1. What is MDC?

One of the most important logging concepts in modern enterprise applications.

---

# Problem

Consider a Spring Boot application handling multiple requests simultaneously.

Example:

```text
Request A → Create Employee

Request B → Update Employee

Request C → Delete Employee
```

---

Generated Logs:

```text
INFO Employee Created

INFO Employee Updated

INFO Employee Deleted
```

---

Question:

```text
Which log belongs to which request?
```

---

Answer:

```text
Impossible To Determine
```

without additional context.

---

# Solution

Use:

```text
MDC (Mapped Diagnostic Context)
```

---

## Definition

> MDC (Mapped Diagnostic Context) is a feature provided by SLF4J and Logback that allows contextual information to be attached to log messages and automatically included in every log generated by the current thread.

---

# Simple Definition

MDC allows us to attach:

```text
Request ID

User ID

Transaction ID

Session ID

Correlation ID
```

to every log statement.

---

# Example

Without MDC:

```text
INFO Employee Created

INFO Employee Updated
```

---

With MDC:

```text
INFO [REQ-1001]
Employee Created

INFO [REQ-1002]
Employee Updated
```

---

Now logs can be traced easily.

---

# Why Is It Called Mapped Diagnostic Context?

---

## Mapped

Stores data as:

```text
Key → Value
```

pairs.

---

Example:

```java
MDC.put(
    "requestId",
    "REQ-1001"
);
```

---

Internally:

```text
requestId → REQ-1001
```

---

## Diagnostic

Used for:

```text
Debugging

Troubleshooting

Monitoring
```

---

## Context

Additional information associated with a request or transaction.

---

# Internal Structure

```text
Thread

   ↓

MDC Map

   ↓

requestId → REQ-1001

userId → 100

transactionId → TXN-500
```

---

# 3. How MDC Works Internally

Important interview topic.

---

# Step 1

Application receives request.

---

Example:

```text
Request ID = REQ-1001
```

---

# Step 2

Store value in MDC.

```java
MDC.put(
    "requestId",
    "REQ-1001"
);
```

---

# Step 3

Logger generates logs.

```java
log.info("Employee Created");
```

---

# Step 4

Logback reads MDC values.

---

# Step 5

MDC values are appended to logs.

---

Generated Output:

```text
[REQ-1001]
Employee Created
```

---

# Internal Flow

```text
Request
     ↓
MDC.put()
     ↓
Logger
     ↓
Logback
     ↓
Read MDC Values
     ↓
Write Log
```

---

# Thread Local Storage

MDC internally uses:

```text
ThreadLocal
```

---

Meaning:

```text
Each Thread Has Its Own MDC Data
```

---

Example:

```text
Thread 1

requestId=REQ-1001
```

---

```text
Thread 2

requestId=REQ-1002
```

---

No interference occurs.

---

# Architecture

```text
Thread

   ↓

ThreadLocal

   ↓

MDC Map

   ↓

Logger
```

# 4. Request Correlation

One of the most common MDC use cases.

---

## What is Request Correlation?

Request Correlation means:

```text
Identifying All Logs
Generated By A Single Request
```

---

# Problem

Request Flow:

```text
Controller

Service

Repository
```

---

Logs:

```text
Controller Started

Employee Saved

Database Updated
```

---

Question:

```text
Which request generated them?
```

---

# Solution

Assign a unique:

```text
Request ID
```

---

Example

```java
String requestId =
UUID.randomUUID().toString();

MDC.put(
    "requestId",
    requestId
);
```

---

Generated Logs

```text
[REQ-1001]
Controller Started

[REQ-1001]
Employee Saved

[REQ-1001]
Database Updated
```

---

Now all logs can be grouped.

---

# Request Correlation Flow

```text
Incoming Request

      ↓

Generate Request ID

      ↓

Store In MDC

      ↓

Application Logs

      ↓

Request ID Added
```

---

# Benefits

```text
Trace Complete Request

Debug Faster

Simplify Monitoring
```

# 7. Using MDC in Spring Boot

Most commonly used methods.

---

## Add Data

```java
MDC.put(
    "requestId",
    requestId
);
```

---

Stores:

```text
requestId → value
```

---

## Read Data

```java
MDC.get(
    "requestId"
);
```

---

Returns:

```text
Stored Value
```

---

## Remove Single Entry

```java
MDC.remove(
    "requestId"
);
```

---

Removes:

```text
requestId
```

---

## Clear Entire MDC

```java
MDC.clear();
```

---

Removes:

```text
All Context Data
```

---

# Important

Always clear MDC after request processing.

---

Bad:

```java
MDC.put(
    "requestId",
    requestId
);
```

without cleanup.

---

Risk:

```text
Data Leakage Between Requests
```

---

Good:

```java
try {

    MDC.put(
        "requestId",
        requestId
    );

}
finally {

    MDC.clear();

}
```

# 8. Log Pattern Configuration

MDC values appear only if included in the log pattern.

---

# Example

```xml
<pattern>

%d [%thread]
[%X{requestId}]
%-5level
%logger
-
%msg%n

</pattern>
```

---

# Meaning

```text
%X{requestId}
```

reads value from MDC.

---

# Example Output

```text
2026-06-15

[main]

[REQ-1001]

INFO

EmployeeService

-

Employee Created
```

---

# Multiple MDC Fields

Pattern:

```xml
%X{requestId}
%X{userId}
%X{transactionId}
```

---

Output:

```text
[REQ-1001]
[USER-500]
[TXN-1000]
```

# 11. Best Practices

---

## Always Use Request ID

Recommended:

```text
Every Incoming Request
```

---

## Include MDC Fields In Pattern

Otherwise:

```text
Values Won't Appear
```

---

## Clear MDC

Always:

```java
MDC.clear();
```

---

## Use Correlation IDs

Useful for:

```text
Microservices
```

---

## Avoid Sensitive Data

Never store:

```text
Passwords

Tokens

Credit Cards
```

# 12. Common Mistakes

---

Problem:

```text
Context Leakage
```

---

Problem:

```text
Values Stored

But Not Displayed
```

---

## Storing Sensitive Data

Bad:

```java
    "password",

);
```

Never do this.
